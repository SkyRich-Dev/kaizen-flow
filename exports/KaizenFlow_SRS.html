1. Introduction

1.1 Purpose

This Software Requirements Specification (SRS) document describes the functional and non-functional requirements for the KaizenFlow Risk Assessment & Approval System. It is intended for stakeholders, developers, testers, and project managers involved in the development and deployment of the system.

1.2 Scope

KaizenFlow is an enterprise web application designed to digitize and automate the Kaizen Risk Assessment Process. The system manages improvement requests through a multi-stage approval workflow involving 5 departments, with cost-based escalation to senior management.

2. Overall Description

2.1 Product Perspective

KaizenFlow is a standalone web application that replaces the existing paper-based Kaizen approval process. It integrates with:

  • PostgreSQL database for data persistence
  • Email servers for notifications
  • WhatsApp Business API for mobile notifications

2.2 Product Functions

  • User authentication and role-based access control
  • Kaizen request creation and management
  • Multi-stage approval workflow automation
  • Cross-department risk evaluation
  • Cost-based approval routing
  • Real-time notifications
  • Comprehensive reporting and analytics
  • Complete audit trail

3. Functional Requirements

3.1 User Management

FR-001 User Authentication

Priority: High

The system shall authenticate users using email and password credentials.

  • Users shall be able to log in with email and password
  • Passwords shall be hashed using bcrypt algorithm
  • Session shall be managed using JWT tokens
  • Tokens shall expire after 24 hours

FR-002 Role-Based Access Control

Priority: High

The system shall restrict access based on user roles.

  • 6 roles: INITIATOR, MANAGER, HOD, AGM, GM, ADMIN
  • Each role has predefined permissions
  • Users can have only one role
  • Users are assigned to one department

FR-003 User Profile Management

Priority: Medium

Administrators shall be able to create, update, and deactivate user accounts.

FR-011 Request ID Generation

Priority: High

The system shall automatically generate unique request IDs in format: KZN-YYYY-NNNN

  • YYYY = Current year
  • NNNN = Sequential 4-digit number, zero-padded

FR-012 Draft Saving

Priority: Medium

Users shall be able to save requests as drafts without submitting for approval.

FR-013 File Attachments

Priority: Medium

Users shall be able to attach files to requests.

  • Supported categories: PFMEA, Check Sheet, CRR, Photos, Other
  • Maximum file size: 10MB per file
  • Maximum 10 attachments per request

3.3 Approval Workflow

FR-020 Own Manager Approval

Priority: High

When a request is submitted, it shall be routed to the initiator's department manager.

  • Manager can approve or reject
  • Rejection requires a reason
  • Comments are optional

FR-021 Own HOD Approval

Priority: High

After manager approval, the request shall be routed to the department HOD.

FR-022 Cross-Department Manager Evaluation

Priority: High

After HOD approval, ALL managers from OTHER departments shall evaluate the request.

  • 4 manager evaluations required (5 departments - own department)
  • Each manager answers department-specific questions
  • Risk levels must be assigned (Low/Medium/High)
  • Remarks mandatory for "No" answers or "High" risk

FR-023 Cross-Department HOD Evaluation

Priority: High

After all managers accept, ALL HODs from OTHER departments shall evaluate.

FR-024 Cost-Based Escalation

Priority: High

The system shall automatically escalate based on cost thresholds:

ConditionEscalation
Cost > ₹50,000Route to AGM
Cost > ₹100,000Route to GM after AGM
Process Addition = trueRoute to AGM
Manpower Addition = trueRoute to AGM

FR-025 AGM Approval

Priority: High

AGM shall review requests that meet escalation criteria.

FR-026 GM Approval

Priority: High

GM shall provide final approval for requests exceeding ₹100,000.

4. Non-Functional Requirements

4.1 Performance

NFR-001 Response Time

Page load time shall not exceed 3 seconds for 90% of requests under normal load.

NFR-002 Concurrent Users

The system shall support minimum 100 concurrent users without degradation.

NFR-003 Database Performance

Database queries shall complete within 500ms for 95% of requests.

4.2 Security

NFR-010 Data Encryption

All data in transit shall be encrypted using TLS 1.2 or higher.

NFR-011 Password Security

Passwords shall be hashed using bcrypt with minimum 10 rounds.

NFR-012 Session Security

JWT tokens shall expire after 24 hours and include user role claims.

NFR-013 Input Validation

All user inputs shall be validated and sanitized to prevent injection attacks.

4.3 Reliability

NFR-020 Availability

The system shall maintain 99.5% uptime during business hours (8 AM - 8 PM IST).

NFR-021 Data Backup

Database backups shall be performed daily with 30-day retention.

NFR-022 Error Handling

The system shall gracefully handle errors and display user-friendly messages.

4.4 Usability

NFR-030 Responsive Design

The interface shall be usable on devices with screen widths from 320px to 2560px.

NFR-031 Browser Compatibility

The system shall work on Chrome, Firefox, Edge, and Safari (latest 2 versions).

NFR-032 Accessibility

The system shall comply with WCAG 2.1 Level AA guidelines.

4.5 Maintainability

NFR-040 Code Standards

Code shall follow industry best practices and be properly documented.

NFR-041 Logging

Application logs shall be maintained for debugging and monitoring.
